# Аудит БД и схемы (2026-02-20)

## Область проверки
- `lib/data/db/schema.ts` (актуальная схема Drizzle)
- `lib/data/db/migrations/*.sql` + `meta/_journal.json` (история миграций)
- ключевые query-слои в `lib/data/db/*-queries.ts` и `lib/infrastructure/auth/rbac.ts`

## Итог
Обнаружены архитектурные и эксплуатационные риски в целостности данных, миграционной дисциплине и RBAC-согласованности.

## Критичные замечания

### 1) Drift схемы и миграций (несинхронный state для Drizzle)
**Симптом:** при `pnpm db:generate` генерируется новая миграция с уже существующими объектами (таблицы/индексы/enum), хотя они уже присутствуют в предыдущих SQL-файлах.

**Риск:**
- ложные диффы и «шумные» миграции;
- рост вероятности конфликтов при релизе;
- ухудшение предсказуемости CI/CD.

**Вероятная причина:** часть миграций писалась вручную с `IF NOT EXISTS`/`DO ... duplicate_object`, и это не полностью отражено в snapshot-истории Drizzle (`meta`).

**Рекомендации:**
1. Провести одноразовую нормализацию: привести `schema.ts`, SQL-миграции и `meta` к единому состоянию (baseline).
2. Для будущих изменений использовать единый поток: `schema.ts` -> `db:generate` -> review SQL -> commit.
3. Запретить ручные «идемпотентные» DDL-миграции без обновления snapshot.

### 2) Ослабленная tenant-целостность между связанными таблицами
**Симптом:** во многих дочерних таблицах есть `tenant_id`, но внешние ключи ссылаются только на `id` родителя (без проверки пары `(id, tenant_id)`). Пример: `estimate_rows.estimate_id -> estimates.id`.

**Риск:**
- на уровне БД возможно вставить запись с `tenant_id = A`, ссылающуюся на сущность tenant `B`;
- потенциальная cross-tenant утечка при ошибке в сервисном слое.

**Рекомендации:**
1. Добавить составные ограничения/ссылки: `(estimate_id, tenant_id) -> estimates(id, tenant_id)` и аналогично для остальных tenant-таблиц.
2. Для ключевых связей внедрить CHECK/trigger-валидации tenant-согласованности.

## Высокий приоритет

### 3) Некорректная «активная» уникальность в `team_members`
Индекс `team_members_active_unique` объявлен как уникальный по `(team_id, user_id)`, но не фильтрует `left_at IS NULL`.

**Риск:**
- повторное вступление пользователя в команду после выхода может быть заблокировано,
- невозможность корректно хранить историю membership.

**Рекомендация:** заменить на partial unique index:
- `UNIQUE(team_id, user_id) WHERE left_at IS NULL`.

### 4) Несогласованность ролевой модели
В проектных требованиях RBAC фигурируют роли `owner`, `admin`, `member`, `estimator`, `manager`, но `tenant_role` ограничен значениями `admin|estimator|manager`.

**Риск:**
- несовпадение бизнес-правил и ограничений БД;
- возможные runtime-ошибки при расширении прав или миграции данных.

**Рекомендация:** синхронизировать матрицу ролей между БД enum, policy-слоем и UI/валидацией.

### 5) Денежные поля на `double precision`
Суммы/цены/количества во многих таблицах хранятся в `double precision`.

**Риск:**
- ошибки округления и накопленные расхождения в финансовых расчётах.

**Рекомендация:** критичные денежные поля перевести на `numeric(14,2)`/`numeric(18,4)` (в зависимости от домена), с миграцией и регрессионными тестами.

## Средний приоритет

### 6) Непоследовательная политика soft-delete в уникальных индексах
Часть таблиц использует partial unique по `deleted_at IS NULL`, часть — глобальные unique без учёта soft-delete.

**Риск:**
- невозможность переиспользования бизнес-ключа после soft-delete (где это ожидалось).

**Рекомендация:** принять единый стандарт по таблицам (что уникально «навсегда», а что только среди активных записей).

### 7) Админские метрики могут включать soft-deleted записи
В `admin-queries.ts` подсчёты по `works/materials` идут только по `tenant_id`, без фильтра `deleted_at IS NULL`.

**Риск:**
- искажённые показатели в админке.

**Рекомендация:** применить `withActiveTenant` (если применимо по контексту) или минимум добавить `isNull(deletedAt)`.
