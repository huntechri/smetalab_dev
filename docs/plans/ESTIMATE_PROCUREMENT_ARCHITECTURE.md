# Закупки в табе сметы (не глобальные): архитектура и логика

## 1) Цель компонента

Реализовать таб `Закупки` внутри конкретной сметы, который:

1. Показывает **только материалы** из текущей сметы.
2. Консолидирует одинаковые материалы по `materialId` (даже если они находятся в разных работах).
3. Показывает рядом фактические данные закупок из модуля глобальных закупок: количество, среднюю цену, сумму.
4. Дает быстрый контроль отклонений: план vs факт.
5. Обязательно показывает материалы, которые были фактически закуплены, но отсутствуют в смете (внизу списка отдельным блоком/сортировкой).

---

## 2) Границы и терминология

- **План** = материалы из `estimate_rows` c `kind = material` внутри конкретной сметы.
- **Факт** = строки из глобальных закупок, связанные с этим же материалом.
- **Не трогаем работы** в UI/агрегации этого таба (работы — только источник материалов).
- Это **не отдельный модуль глобальных закупок**, а аналитический срез внутри сметы.

## 4) Модель данных для таба

Рекомендуемый DTO:

```ts
export type EstimateProcurementRow = {
  materialId: string;
  materialName: string;
  unit: string;
  source: 'estimate' | 'fact_only';

  // План из сметы
  plannedQty: number;
  plannedPrice: number; // средневзвешенная плановая цена по смете
  plannedAmount: number;

  // Факт из глобальных закупок
  actualQty: number;
  actualAvgPrice: number; // средневзвешенная фактическая цена
  actualAmount: number;

  // Производные
  qtyDelta: number; // plannedQty - actualQty
  amountDelta: number; // plannedAmount - actualAmount

  // Диагностика
  purchaseCount: number;
  lastPurchaseDate: string | null;
};
```

> Важно: фактическая средняя цена считается как **взвешенная** по количеству, а не как простое среднее по чекам.

> Важно: `source = 'fact_only'` означает, что позиция отсутствует в смете, но присутствует в фактических закупках, и должна попадать в таблицу в конце списка.

## 5) Алгоритм консолидации

## 5.1 Шаг A: собрать план из сметы

Источник: `estimate_rows`.

Условия:

- `estimateId = текущая смета`
- `kind = 'material'`
- `deletedAt IS NULL`
- в tenant scope через `withActiveTenant(...)`

Группировка:

- ключ: `materialId`
- `plannedQty = SUM(qty)`
- `plannedAmount = SUM(qty * price)`
- `plannedPrice = plannedAmount / plannedQty`, если `plannedQty > 0`, иначе `0`

## 5.2 Шаг B: собрать факт из глобальных закупок

Источник: таблица(ы) глобальных закупок (материалы закупок).

Условия:

- tenant scope через `withActiveTenant(...)`
- учитывать только активные строки (`deletedAt IS NULL`)
- join по `materialId` с материалами из шага A

Группировка:

- ключ: `materialId`
- `actualQty = SUM(purchaseQty)`
- `actualAmount = SUM(purchaseQty * purchasePrice)`
- `actualAvgPrice = actualAmount / actualQty`, если `actualQty > 0`, иначе `0`
- `purchaseCount = COUNT(*)`
- `lastPurchaseDate = MAX(purchaseDate)`

## 5.3 Шаг C: мердж план/факт

Для каждого `materialId` из плана:

- подтянуть факт (если нет — нули)
- посчитать дельты

Обязательно дополнительно включить материалы, которые есть в факте, но нет в смете:

- `plannedQty = 0`
- `plannedPrice = 0`
- `plannedAmount = 0`
- `source = 'fact_only'`

Рекомендованная сортировка по умолчанию:

1. Сначала `source = 'estimate'`
2. Затем `source = 'fact_only'`
3. Внутри групп — по `materialName` (ASC)

В UI добавить быстрый фильтр:

- «Только вне сметы» (`source = 'fact_only'`)

## 6) Формула средней цены (критичный момент)

Если закупки:

- 50 шт × 100
- 30 шт × 120
- 20 шт × 130

Тогда:

- `actualAmount = 5000 + 3600 + 2600 = 11200`
- `actualQty = 100`
- `actualAvgPrice = 11200 / 100 = 112`

То есть нельзя брать `(100 + 120 + 130)/3`.

---

## 7) Валидации и ограничения

1. Если у строки сметы нет `materialId` — такая строка не может корректно консолидироваться. Рекомендация: показывать блок «неконсолидационные материалы» с предупреждением.
2. Если единица измерения у одного `materialId` различается, показывать warning badge `unit mismatch`.
3. Денежные значения хранить в числах БД, а форматировать в UI (`Intl.NumberFormat`).
4. Нужна единая точность округления (например, 2 знака после запятой) только на уровне отображения.

## 8) Производительность (критично для тяжелого таба)

1. Индексы:
   - `estimate_rows(tenant_id, estimate_id, kind) WHERE deleted_at IS NULL`
   - `global_purchase_items(tenant_id, material_id, purchase_date) WHERE deleted_at IS NULL`
2. Сначала получить список `materialId` из сметы, затем ограничить факт `IN (...)`.
3. Для больших смет: серверная пагинация/виртуализация таблицы.
4. Кэшировать результат на короткое время (например, 30–60 сек), инвалидировать после изменений закупок/строк сметы.

Дополнительно (рекомендуется для production-нагрузки):

5. **Двухфазная агрегация на SQL-уровне**:
   - CTE `plan_agg` (агрегат сметы по `materialId`)
   - CTE `fact_agg` (агрегат закупок по `materialId`, ограниченный `materialId` из `plan_agg` + отдельный проход для `fact_only`)
   - `FULL OUTER JOIN`/`UNION ALL` для формирования полного набора строк.
6. **Серверная пагинация + стабильная сортировка** вместо передачи всех строк в клиент для больших смет.
7. **Избежать N+1**: не делать отдельный запрос на каждый материал для факта.
8. **Дешевый режим таба по умолчанию**:
   - первично грузить только агрегированные поля;
   - детализацию закупок (даты/история) подгружать по раскрытию строки (lazy).
9. **Кэш-ключ уровня сметы**: `tenantId + estimateId + filters + updatedAtCursor`.
10. **Инвалидация кэша событиями**:
    - обновление строк сметы (materials);
    - создание/изменение/удаление строк глобальной закупки по связанным materialId.
11. **Числовая стабильность**:
    - расчеты выполнять на сервере в decimal/numeric;
    - округление только на уровне UI.
12. **Лимиты и guardrails**:
    - hard-limit на размер `IN (...)` батча (например, чанки по 500–1000 id);
    - timeout-safe стратегия (возврат частичных данных с индикатором, если SLА превышен).
13. **Observability**:
    - метрики: latency p50/p95, count rows, cache hit-rate;
    - логирование тяжелых запросов (`EXPLAIN ANALYZE` в профилировании).

## 11) Что может быть нелогично/рискованно

1. **Простое среднее цены** — математически неверно, приведет к искажению бюджета.
2. **Связка по имени материала** вместо `materialId` — высокий риск ошибок из-за дублей/переименований.
3. **Смешение разных единиц измерения** без явной обработки — ломает смысл агрегатов.
4. **Запрос без tenant helper** — риск утечки данных между компаниями.
5. **Терять fact_only-позиции** — пользователи не увидят закупки вне сметы и получат ложную картину исполнения бюджета.
