# UI v2 — чистая пересборка интерфейса на shadcn/ui

## Статус

Черновой архитектурный контракт.

Этот документ описывает правила новой UI v2-архитектуры для SmetaLab.

UI v2 — это не рефакторинг старого UI.  
UI v2 — это параллельная чистая пересборка экранов по одному.

## Зачем нужен UI v2

Текущий UI стал слишком запутанным:

- слишком много shared wrappers;
- слишком много UI ownership правил;
- слишком много исторических compatibility-слоёв;
- слишком много связей между экранами и старым `shared/ui`;
- рефакторинг больше не даёт достаточной ясности.

Цель UI v2 — построить интерфейс заново, проще и понятнее.

## Главный принцип

Сначала верстаем экран на mock data.

Backend подключаем только после того, как UI/UX экрана утверждён.

## Что UI v2 не делает

UI v2 не должен:

- рефакторить старый UI;
- использовать старый `shared/ui`;
- переписывать backend;
- менять базу данных;
- менять server actions;
- создавать новую кастомную дизайн-систему;
- создавать новые универсальные wrappers;
- создавать аналоги `Surface`, `CardShell`, `DataTableShell`, `PageShell`.

## Источник правды

Все правила UI v2 находятся в:

```txt
docs/ui-v2/**
```

Старые архитектурные документы остаются актуальными для legacy-кода, но новый UI v2 живёт по этим правилам.

## Основное правило

UI v2 не должен импортировать legacy UI.

Запрещено в UI v2:

```txt
@/shared/ui/*
@repo/ui
@/features/_shared/*
старые shell wrappers
старые table wrappers
старые surface/card wrappers
старые feature wrappers
server actions
DB imports
lib/data
lib/domain
lib/services
```

Разрешено в UI v2:

```txt
@/components/ui/*
@/components/app-shell/*
@/features-v2/*
локальные mock data
локальные feature types
@/lib/utils только для cn при необходимости
```

## Стратегия пересборки

Каждый экран пересобирается отдельно.

Порядок для каждого экрана:

1. Описать экран.
2. Определить нужные shadcn/ui primitives.
3. Подготовить mock data.
4. Сверстать статичный UI.
5. Проверить desktop layout.
6. Проверить mobile layout.
7. Добавить empty state.
8. Добавить loading skeleton.
9. Добавить dialogs/sheets, если нужны.
10. Утвердить UX.
11. Подключить backend.
12. Заменить legacy route.
13. Удалить старый код экрана.

## Первый экран

Первый экран для UI v2:

```txt
Projects List
```

Не начинать со страницы сметы.

Страница сметы слишком сложная для первого прохода: вложенные работы/материалы, inline edit, коэффициенты, выполнение, закупки, агрегации, будущий backend.

## Definition of Done для документационного этапа

Этап документации завершён, если:

- не созданы `.tsx` файлы;
- не изменён runtime code;
- не изменён backend;
- не изменена база данных;
- не изменён `components.json`;
- не изменён ESLint;
- описана структура UI v2;
- описаны разрешённые и запрещённые импорты;
- описана component policy;
- описана карта экранов;
- описан план миграции;
- выбран первый экран.
